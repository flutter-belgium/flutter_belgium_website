# Made in Flutter Belgium uses slug-based URLs derived from names

## Context and Problem Statement

Each app, company, and developer needs a stable, human-readable URL for their detail page. The data API identifies entries by name (apps, companies) or GitHub username (developers). A URL strategy is needed.

## Decision Drivers

- URLs should be human-readable and shareable
- Slugs must be deterministic from the name so routes can be generated at build time without a separate slug field in the data
- Developer URLs should be consistent with how developers are identified across the web (GitHub username)

## Considered Options

- Numeric IDs (e.g. `/apps/42`) — opaque, not human-readable
- Raw name in URL (e.g. `/apps/My App!`) — URL encoding issues, fragile
- Slugified name (e.g. `/apps/my-app`) — human-readable, URL-safe, deterministic

## Decision Outcome

Chosen option: **Slugified name**, generated by `toSlug()` in `lib/util/made_in_utils.dart`:

```dart
String toSlug(String name) {
  return name
      .toLowerCase()
      .replaceAll(RegExp(r'[^a-z0-9\s]'), '')
      .replaceAll(RegExp(r'\s+'), '-');
}
```

Applied consistently to all three types:
- Apps: `/made-in-flutter-belgium/apps/{toSlug(app.name)}`
- Companies: `/made-in-flutter-belgium/companies/{toSlug(company.name)}`
- Developers: `/made-in-flutter-belgium/developers/{toSlug(developer.githubUserName)}`

### Consequences

- Good, because URLs are human-readable and shareable
- Good, because slug generation is deterministic — no database or separate slug field needed
- Good, because `toSlug()` is used consistently in both routing and link generation
- Bad, because renaming an app or company in the data source changes its URL, breaking existing links — acceptable given the early stage of the feature
